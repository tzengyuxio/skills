# Humanize — 示範

展示 `humanize` skill 如何辨識並改寫 AI 寫作痕跡。本 skill 沒有強度分層，採「Before / After」對照。

> **源文說明**：本 demo 的 Before 段落為示範用 AI 風格範本（自編），不放入共用 [`source-texts/`](../../examples/source-texts/) — 因為「有問題的文本」不適合作為他用。

---

## 範例一：時事評論段落

### Before（AI 風格）

> 在當今這個快速變化的時代，**人工智慧的崛起**不僅深刻改變了我們的工作方式，更重塑了整個社會的運作邏輯。許多專家認為，AI 將會帶來前所未有的機遇，同時也伴隨著值得關注的挑戰。從教育到醫療，從金融到製造業，沒有一個行業能夠免於這場變革的影響。展望未來，如何在擁抱技術的同時保持人的價值，將是我們這一代人必須認真思考的重要課題。

**問題標記：**

| 編號 | 模式 | 出處 |
|---|---|---|
| C1 | 宏大化開場 | 「在當今這個快速變化的時代」 |
| C3 | 模糊歸因 | 「許多專家認為」（沒名沒姓） |
| C5 | 宣傳語氣 | 「前所未有的機遇」 |
| V1 | 簽名詞彙 | 「不僅…更…」「值得關注」「展望未來」 |
| S1 | 三元排比 | 「從教育到醫療，從金融到製造業」 |
| S3 | 公式化結尾 | 「這一代人必須認真思考的重要課題」 |
| C2 | 空洞總結 | 「保持人的價值」（無具體內容） |

### After（自然中文）

> AI 改變了不少行業的工作流程。最容易看到的是內容創作、客服和程式設計這幾個領域——重複性高的部分，模型現在做得比人快也便宜。其他行業變化沒那麼立竿見影：醫療要等臨床驗證，金融受監管限制，製造業則卡在感測器和資料品質。
>
> 真正的問題不是「AI 會不會取代人類」，而是企業端要決定哪些任務值得交給模型、哪些不值得。這是個工作流程設計題，不是哲學辯論。

**改寫重點：**

- 拿掉宏大化開場，第一句直接給結論
- 「許多專家認為」改成可驗證的具體領域
- 用具體例子（內容創作／客服／程式設計）取代抽象排比
- 結尾不展望未來，改提具體的決策框架

## 範例二：產品描述段落

### Before（AI 風格）

> 我們**不僅僅**是一款工具，**更是**一個賦能團隊協作的全方位解決方案。透過先進的人工智慧技術，本產品為使用者提供了前所未有的順暢體驗。無論是個人開發者、新創團隊、還是大型企業，都能從中受益。我們致力於打造一個值得信賴的平台，讓每一位使用者都能專注於真正重要的事情。

### After（自然中文）

> 這是一款給開發團隊用的協作工具。核心功能有三個：共用程式碼空間、即時討論、版本紀錄。和同類工具的差別主要在 AI 自動補全——它會讀整個 repo 的上下文，不只是當前檔案。
>
> 適合 5 到 50 人的工程團隊；超過這個規模，建議搭配企業版的權限管理。

**改寫重點：**

- 「不僅僅…更是…」「賦能」「全方位解決方案」全屬 V1 簽名詞彙，刪掉
- 「先進」「前所未有」「值得信賴」屬 C5 空洞讚美，刪掉
- 「個人開發者、新創團隊、大型企業」屬 S8 假範圍，改為具體規模上限
- 「致力於」「真正重要的事情」屬 C2 空洞總結，刪掉

## 範例三：開頭段落（C1 + S6）

### Before

> 隨著科技的飛速發展，遠距工作已經成為一種**新常態**——它不僅打破了地理的限制，**更重新定義**了人們對工作與生活平衡的理解。

### After

> 遠距工作從疫情期間的應急方案，慢慢變成一部分公司的長期選項。它解決了通勤時間的問題，但也讓「下班」的界線變得模糊。

## 改寫原則速查

13 條原則中最常用的：

| 原則 | 怎麼用 |
|---|---|
| 1. 具體取代籠統 | 「許多企業」→「Google、Stripe、Cloudflare」 |
| 2. 展示而非宣告 | 「強大的功能」→ 列出具體功能 |
| 5. 歸因要點名 | 「專家認為」→ 點名某機構某報告，或刪掉 |
| 7. 允許不完美 | 不每段都拔到「這是一個重要的議題」 |
| 8. 用平常的詞 | 「賦能」→「讓 X 能 Y」；「致力於」→「在做」 |
| 12. 用簡單動詞 | 「進行討論」→「討論」；「作為一個」→ 直接刪 |

完整 26 項 checklist 與 13 條原則見 [SKILL.md](../SKILL.md)。
